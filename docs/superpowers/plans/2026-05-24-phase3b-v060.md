# Phase 3b Implementation Plan (v0.6.0)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Approval queue — agents that trigger `require_approval` policy decisions get their requests stored in a JMAP mailbox for human review. UI surfaces pending approvals with approve/deny/modify actions.

**Architecture:** Approval requests stored as emails in a dedicated JMAP mailbox (role `approvals`). Status tracked via keywords (`$approval_pending`, `$approval_approved`, `$approval_denied`). MCP server creates approval entries when policy returns `require_approval`. Webmail shell renders an Approvals view with badge count. Race condition safety via re-dry-run on approve.

**Tech Stack:** TypeScript, JMAP (Email/set, Email/query), React, Jotai, Vitest

**Spec:** `docs/superpowers/specs/2026-05-23-phase3-design.md` — Cut 3b (items 5, 6, 7)

### Task 1: Approval Store (JMAP Mailbox Backend) — Item 5

**Files:**
- Create: `shell/src/runtime/approval-store.ts`
- Create: `shell/src/runtime/__tests__/approval-store.test.ts`
- Modify: `shell/src/runtime/jmap-client.ts` (add Mailbox/set create, Email/set for approvals)

**Interface:**
```typescript
export type ApprovalRequest = {
  readonly id?: string; // JMAP email ID (set after creation)
  readonly schemaVersion: 1;
  readonly toolName: string;
  readonly requestingAgentId: string;
  readonly requestingAgentName: string;
  readonly params: unknown;
  readonly preview: unknown;
  readonly previewHashHex: string;
  readonly requestedAt: string;
  readonly status: 'pending' | 'approved' | 'denied';
};

export interface ApprovalStore {
  ensureMailbox(session: Session): Promise<string>; // returns mailbox ID
  create(session: Session, request: Omit<ApprovalRequest, 'id' | 'status'>): Promise<string>; // returns email ID
  list(session: Session, filter?: { status?: string }): Promise<readonly ApprovalRequest[]>;
  approve(session: Session, approvalId: string): Promise<void>;
  deny(session: Session, approvalId: string): Promise<void>;
  get(session: Session, approvalId: string): Promise<ApprovalRequest | null>;
}
```

**JMAP implementation:**
- `ensureMailbox`: Email/query for mailbox with name "Approvals", create via Mailbox/set if absent
- `create`: Email/set create with `$approval_pending` keyword, body = JSON payload
- `list`: Email/query in approvals mailbox filtered by keyword, Email/get for bodies
- `approve`/`deny`: Email/set update to toggle keywords
- `get`: Email/get by ID, parse body JSON

**Tests:** In-memory mock implementation for unit tests. Test create→list round-trip, status transitions, filter by status.

### Task 2: Policy Engine v2 (Wire require_approval) — Item 5 (continued)

**Files:**
- Modify: `shell/src/runtime/policy-engine.ts` (add configurable policy rules)
- Modify: `mcp-server/src/http-transport.ts` (on require_approval → create approval entry → return pending_approval response)

**Policy engine v2:** Add a `configurablePolicyEngine` that accepts rules:
```typescript
type PolicyRule = {
  readonly toolName: string;
  readonly decision: 'allow' | 'deny' | 'require_approval';
  readonly reason?: string;
};
```

Default rules: all destructive tools from non-UI callers → `require_approval`. Read tools → `allow`.

**MCP integration:** When policy returns `require_approval`:
1. Create approval entry via ApprovalStore
2. Return `{ status: "pending_approval", approvalId, message }` to the agent
3. Agent can poll `GET /agents/approvals/{id}` for status

**Add poll endpoint:** `GET /agents/approvals/{id}` — returns approval status + committed result when approved.

### Task 3: Approvals View UI — Item 6

**Files:**
- Create: `shell/src/views/approvals-view.tsx`
- Create: `shell/src/views/__tests__/approvals-view.test.tsx`
- Modify: `shell/src/App.tsx` (enable Approvals nav tab, wire to ApprovalStore, add badge count)

**Layout:** Single-column list of pending approvals. Each card:
- Agent name, tool name, summary, timestamp
- Expandable preview (rendered readably, not raw JSON)
- Approve / Deny / Modify & Approve buttons

**Approve flow:**
1. Re-run dry-run with original params
2. If preview matches → commit → set `$approval_approved`
3. If preview differs → show diff → require re-confirmation

**Badge:** Count of `$approval_pending` items shown on the Approvals tab.

**Polling:** Refresh pending count every 30s (push replaces this in 3c).

### Task 4: Approval Notification Path — Item 7

**Files:**
- Modify: `mcp-server/src/http-transport.ts` (add `GET /agents/approvals/{id}` poll endpoint)
- Modify: agent response shape for `require_approval`

**Agent side:** MCP response on `require_approval`:
```json
{
  "status": "pending_approval",
  "approvalId": "<email-id>",
  "message": "This action requires human approval.",
  "pollUrl": "/agents/approvals/{approvalId}"
}
```

**Poll endpoint:** `GET /agents/approvals/{id}` returns:
- `{ status: "pending" }` — still waiting
- `{ status: "approved", result: <committed output> }` — approved and committed
- `{ status: "denied", reason: "..." }` — denied
