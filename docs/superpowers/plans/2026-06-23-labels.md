# Labels (P1.1) Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Gmail-style labels — tag messages, manage label definitions (create/rename/recolor/reorder/delete), colored chips, sidebar list + filter — for humans and agents at parity, with definitions in a server-side FileNode registry.

**Architecture:** A label is `{ key, name, color, order }`. The JMAP keyword on a message *is* the immutable human-readable `key` (slug minted once from `name`); presentation (`name`/`color`/`order`) lives in a JSON **registry document** stored as one FileNode (`urn:ietf:params:jmap:filenode`) whose content is a blob (`urn:ietf:params:jmap:blob`). Rename/recolor = registry-only edits (no message rewrites). New codegen contracts `label.list/create/update/delete/apply` generate TS types, React hooks, MCP tools, and docs from one source.

**Tech Stack:** TypeScript + React + Vite (`shell/`), Zod codegen (`tools/codegen/`), JMAP over `POST /jmap/`, Vitest.

## Global Constraints

- Label key regex: `^[a-z0-9][a-z0-9_-]{0,62}$` (charset-safe, ≤63 chars), case-insensitively unique, **immutable after creation**.
- Registry cap: **200** labels.
- Default label color: `#ff6b35` (orange accent); palette is a small fixed set.
- Non-destructive **write** caps (`label.create`, `label.update`) are invoked imperatively via `invoker.invoke(...)` — never the generated read-hook (mutate-on-render footgun). Bump `pushGenerationAtom` on success.
- Destructive caps (`label.delete`, `label.apply`) declare `dryRun.preview` with `{ affectedCount: number }` and honor `_options.dryRun`.
- `label.delete` untags messages with a **paging loop** over all matches — no message-count cap (the `mailbox.delete` regression).
- Refusals are typed `code` + the exact human messages in the spec's "Errors & refusals" table — reproduced per task and in `docs/labels.md` and contract `errors`.
- Scopes: `mail:label:read` (list + apply-resolve), `mail:label:write` (create/update/delete); `label.apply` also requires `mail:modify`.
- Follow existing patterns: contracts mirror `tools/codegen/contracts/mailbox-*.ts`; paging untag mirrors `fetchMailboxDeleteCommit`; dialogs mirror `shell/src/components/folder-dialogs.tsx`; picker mirrors the Folders Move-to `MenuButton`.
- Spec: `docs/superpowers/specs/2026-06-23-labels-design.md`.

```ts
const KEY_RE = /^[a-z0-9][a-z0-9_-]{0,62}$/;

function slugify(name: string): string {
  return name
    .toLowerCase()
    .replace(/[^a-z0-9]+/g, '_') // runs of non-alnum → single _
    .replace(/^[_-]+|[_-]+$/g, '') // trim separators
    .slice(0, 63)
    .replace(/[_-]+$/g, ''); // re-trim if slice landed on a separator
}

export function mintLabelKey(
  name: string,
  existingKeys: readonly string[],
): string | null {
  const base = slugify(name);
  if (!base || !KEY_RE.test(base)) return null;
  const taken = new Set(existingKeys.map((k) => k.toLowerCase()));
  if (!taken.has(base)) return base;
  for (let i = 2; ; i++) {
    const candidate = `${base}_${i}`.slice(0, 63).replace(/[_-]+$/g, '');
    if (!taken.has(candidate)) return candidate;
  }
}
```

  serializeRegistry, parseRegistry, EMPTY_REGISTRY, DEFAULT_LABEL_COLOR,

    const r = { version: 1 as const, labels: [
      { key: 'b', name: 'B', color: '#111111', order: 1 },
