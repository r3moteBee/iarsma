# Accessibility — Iarsma policy and tooling

**Last updated:** 2026-05-11
**Status:** baseline locked at Phase 1 (work item 11). Updates ride with PRs that touch a11y-impacting code.

## Target

**WCAG 2.1 Level AA** is the floor. Every UI surface — signed-out, signed-in, modal — must pass the axe-core test suite with zero violations against the WCAG 2.1 AA ruleset.

Component-level accessibility is non-negotiable. Visual / behavioral parity with screen-reader users is a first-class deliverable, not an afterthought. The keyboard model (`docs/keyboard.md`) is the navigation contract; this document is the structural contract.

## Three layers of coverage

### 1. Component-level (Vitest + axe-core)

Every rendered React component has a sibling test file that runs axe-core against the rendered DOM with zero allowed violations:

```ts
import { runAxe } from '../../__tests__/util/axe.js';

it('has zero axe-core violations against WCAG 2.1 AA', async () => {
  const { container } = render(<MyComponent />);
  const violations = await runAxe(container);
  expect(violations.map((v) => v.id)).toEqual([]);
});
```

### 2. Integration-level (Vitest + axe-core)

`shell/src/__tests__/a11y-integration.test.tsx` renders the full 3-column signed-in shell composed together and runs axe over the combined DOM. This catches cross-component violations that per-component tests miss:

- Duplicate landmarks (more than one `<main>`, conflicting `aria-label`s)
- Heading-order regressions (h1 → h3 with no h2)
- ARIA conflicts when components compose (e.g., two regions claiming the same `aria-labelledby`)

The harness markup mirrors `App.tsx`'s `SignedInView` exactly. A drift between the two would surface here as a failing test.

### 3. Built-bundle (pa11y-ci)

`pnpm a11y` builds the production bundle, serves it via `vite preview`, and runs pa11y-ci with the axe runner against the served URLs. This catches a11y issues that only show up in the production build:

- Issues introduced by minification, build-time CSS post-processing, or asset rewriting
- Issues in the unsigned-in entry point that don't depend on JMAP data
- Browser-level a11y checks (color contrast against real computed styles, focus indicators with real rendering)

Configured in `shell/.pa11yci.json`. Runs in CI as the `a11y` job, gating merges alongside `ts` / `rust` / `shell-build`. CI installs Chromium via apt and exports `PUPPETEER_EXECUTABLE_PATH=/usr/bin/chromium-browser`; local devs export the same var pointing at their installed Chromium (`brew install --cask chromium` on macOS, `apt install chromium-browser` on Debian/Ubuntu).

Limitation: pa11y can't sign in to a live mail account in CI, so it only audits surfaces reachable without auth (today: just the sign-in screen). Signed-in surfaces are audited via the integration test layer (#2).

## Manual checks

Automated tooling is necessary but not sufficient. Before each phase ships, the developer runs a manual pass:

- **VoiceOver smoke** (macOS): can a screen-reader user sign in, navigate the mailbox tree, open a thread, and read a message without unannounced state changes?
- **Keyboard-only smoke**: can every interactive surface be reached and operated without a pointer?
- **High-contrast and zoom**: does the UI remain usable at 200% browser zoom and with "Increase Contrast" enabled?

Findings either fix immediately or become tracked issues with `a11y` labels.

## When axe-core flags a violation

The default is **fix the code**. Exceptions are rare and require:

1. A scoped `axe.run` rule override or `disableRules` entry.
2. A code comment at the override site explaining WHY the rule doesn't apply here.
3. A doc entry in this file under "Known exceptions" with the same justification.

Today's known exceptions: **none**.

## Adding a new component

1. Write the component.
2. Write its test file with at least one axe-core block (zero violations against WCAG 2.1 AA).
3. If the component is composed into the signed-in shell, update the integration harness in `a11y-integration.test.tsx` to render it alongside the others.
4. If the component changes the unsigned-in surface, verify `pnpm a11y` still passes locally.

## Tooling versions

- **axe-core**: pinned via `shell/package.json` (`axe-core@^4.x`). Same engine in Vitest and pa11y.
- **pa11y-ci**: pinned via `shell/package.json`. Uses the axe runner (`runners: ['axe']`).
- **WCAG ruleset**: 2.1 Level AA. Bump when the standard does and there's a clear forward-compatibility path.

## Why axe-core in both layers

Using the same engine end-to-end means a violation diagnosed in Vitest is the same violation pa11y would report against the built bundle. The two layers cover different gaps: Vitest catches issues during dev with millisecond feedback; pa11y catches build-pipeline regressions that jsdom can't model.
